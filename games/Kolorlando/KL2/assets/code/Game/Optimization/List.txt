KL2 Optimization List
=====================

Purpose
-------

This file is a working map of KL2 optimizations.

It separates:
- implemented optimizations
- defined but not wired optimization classes
- future optimization plans

Golden Rule
-----------

Core split:
- Data  [Data   "./GoldenRule/Data.js"]
- Mesh  [Mesh   "./GoldenRule/Mesh.js"]
- Render[Render "./GoldenRule/Render.js"]

Rules:
- Data prepares world/chunk data.
- Mesh builds geometry.
- Render decides what is visible and worth drawing.
- GameRuntime coordinates, but should not own low-level optimization logic.

Defined But Not Wired Yet
-------------------------

These files exist as contracts / planning classes.
They are not connected to the active render pipeline yet.

- LOD [LOD "./LOD.js"]
- Impostor [Impostor "./Impostor.js"]
- OcclusionCulling [OcclusionCulling "./Visibility/OcclusionCulling.js"]

Need review before wiring:
- Compare responsibilities.
- Avoid duplicated distance logic.
- Decide which class owns distance bands.
- Decide which class owns cache keys.
- Decide what data goes into worker payloads.

Suggested responsibility split:

LOD:
- Decides detail level.
- Uses 3 modes only:
  - full-detail
  - medium-detail
  - minimal-detail
- Should affect Boxel15 meshes first.
- Later scales down into microxeled voxels.

Impostor:
- Decides when real representation is replaced.
- Should come after LOD.
- Useful for very far land before full mesh streaming.
- Future output could be billboard, boxel card, or color volume.

OcclusionCulling:
- Should only run in caves / closed spaces.
- Should skip open space by default.
- Must stay low priority.
- Must avoid slowing down outdoor traversal.


Future Plan
-----------

Main roadmap:
- LOD tiers for far Boxel15 meshes [LOD "./LOD.js"]
- LOD hysteresis to prevent tier flicker [LOD "./LOD.js"]
- LOD-aware mesh cache per Boxel15 + mode [ChunkVisibility "./Visibility/ChunkVisibility.js"]
- Far-land impostors before full mesh streaming [Impostor "./Impostor.js"]
- Occlusion only for caves / closed spaces [OcclusionCulling "./Visibility/OcclusionCulling.js"]
- Distance bands: near editable, mid greedy mesh, far simplified [Render "./GoldenRule/Render.js"]
- Keep raycast/edit targets near only [Raycast "../../Raycast/Raycast.js"]
- Worker payload includes lodMode / impostorMode [BoxelMeshWorker "./Meshing/BoxelMeshWorker.js"]
- Memory budget for cached far meshes / impostors [RenderBudgeter "./RenderBudgeter.js"]
- Benchmark before enabling each tier [GameRuntime "../GameRuntime.js"]

Execution order:
1. LOD
2. LOD-aware mesh cache
3. Impostors
4. Raycast near-only
5. Occlusion caves-only

LOD detail rule:
- Near: editable, full mesh, full microxels.
- Mid: greedy mesh, simplified microxeled voxels.
- Far: simplified chunk volume or impostor.
- Very far: hidden or fog-covered.

Do not do:
- Do not start LOD at microxel level.
- Do not run occlusion in open space.
- Do not remesh when crossing distance thresholds without hysteresis.
- Do not let GameRuntime become the owner of optimization policy.
- Do not mix LOD and Impostor into the same class too early.


Bottom Notes
------------

LOD.js future use:
- Feed ChunkVisibility with a mode per Boxel15.
- Send lodMode to BoxelMesher / BoxelMeshWorker.
- Cache meshes by boxelKey + lodMode.
- First implementation should only switch Boxel15 mesh detail.
- Later implementation can simplify microxeled voxels internally.

Impostor.js future use:
- Generate far cheap representations.
- Replace full chunk meshes at long distance.
- Should probably use its own future mesher, not BoxelMesher directly.
- Best used after LOD is stable.

OcclusionCulling.js future use:
- Only useful when player is in closed spaces.
- Must detect open-space and skip.
- Should be soft first.
- Hard occlusion can cause popping and should wait.
